# Build Phase

## DEC Learning Hub Course Creator Portal Core Workflow Specification

## A. What the Build Phase Is

The Build Phase is the course production phase of the DEC Learning Hub Course Creator Portal. It begins after the Design-to-Build Handover Record has been approved and converts the approved design into a complete learner-facing course.

This phase uses the platform’s block-based course builder, AI-assisted authoring tools, templates, interactive elements, final tests, downloadable resources, practical output submission, and verified achievement features.

The Build Phase must balance two needs:

- Governed course creation
The course must remain grounded in the approved Analysis and Design records. It should not drift away from the validated CSO capacity gap, baseline, performance goal, approved learning pathway, safeguards, assessment logic, or evaluation anchor.
- Flexible course building
The course creator should still be able to add appropriate blocks during Build when they improve clarity, practice, accessibility, engagement, learner support, or assessment. The Build Phase should not feel like a rigid locked script. It should provide a full block library with all available block types visible in a left-side navigation panel, organized into expandable categories and subcategories.
The key principle is:

The creator may add more blocks, but every added block must be linked to an approved purpose: required action, minimum information, learner practice, final test preparation, practical output, accessibility support, safeguarding note, or learner reflection.

The Analysis Phase provides the evidence base: the validated capacity gap, baseline, standards or indicators, root cause, intervention route, and evaluation anchor. The Design Phase converts that evidence into the action map, blueprint, storyboard, scenario/practice plan, and approved Design-to-Build Handover.

The Build Phase then answers:

How do we turn the approved design into a high-quality, learner-facing course while allowing creators enough flexibility to improve the learning experience through appropriate additional blocks?

## B. Phase 1 Scope Clarification

For Phase 1, the DEC Learning Hub should focus course production on Knowledge and Skill gaps.

Motivation and Environment gaps should not be treated as course-development priorities in Phase 1. They may be recorded, flagged, or routed to other support pathways, but they should not drive Phase 1 course production.

| Diagnosis | Phase 1 Build decision |
| --- | --- |
| Knowledge gap | Build a course, micro-course, explainer, checklist, job aid, knowledge checks, and final test |
| Skill gap | Build guided practice, worked examples, scenarios, template tasks, practical outputs, and final test |
| Motivation gap | Do not build as a Phase 1 course; route outside the Phase 1 course workflow |
| Environment gap | Do not build as a Phase 1 course; route outside the Phase 1 course workflow |
| Mixed gap | Build only the clear Knowledge/Skill component if separable; route Motivation/Environment components outside Phase 1 |

This keeps Phase 1 realistic and focused on producing high-quality self-learning courses, while still preserving evidence about wider capacity challenges for later DEC support.

## C. Why the Build Phase Is Needed

The Build Phase is needed because a good design can still fail during production. A course may become too long, too theoretical, too generic, too media-heavy, or unsafe for the learner context. The Build Phase protects quality while allowing course creators to produce rich, practical, and context-relevant learning experiences.

The Build Phase should ensure that each course:

- follows the approved Analysis and Design logic;
- supports Knowledge and Skill gaps suitable for Phase 1;
- allows creators to add useful blocks without losing traceability;
- remains practical, mobile-first, and low-bandwidth friendly;
- includes appropriate learning activities, examples, checks, templates, and final test items;
- includes practical output or proof submission where relevant;
- supports automated certificate logic;
- supports verified achievement or badge recognition linked to CSO capacity standards;
- remains safe for safeguarding, civic-space, advocacy, and sensitive data contexts;
- is ready for structured Review and eventual Publish.
The Build Phase is therefore not an unrestricted page builder, and it is also not a narrow locked script. It is a guided and flexible block-building workspace: creators can use the full block library, but the platform should require purpose-linking, scope warnings, and readiness checks to keep the course aligned.

## D. Who Does What in the Build Phase

| Actor | Role in the Build Phase |
| --- | --- |
| Course creator / subject matter expert | Builds the course shell, selects blocks, adds or adapts learner-facing content, reviews AI drafts, adds useful supporting blocks, and keeps the course grounded in CSO realities. |
| Instructional design reviewer | Checks whether the course structure, added blocks, practice activities, and assessment flow support the approved design. |
| DEC capacity lead | Confirms that the course still supports the original CSO capacity objective, standard, and performance goal. |
| Safeguarding / civic-space reviewer | Reviews sensitive content, scenarios, data protection issues, advocacy risks, and proof-submission safety. |
| Platform | Displays the locked design reference, provides the full block library, requires block-purpose linking, flags scope drift, manages final test/certificate logic, and tracks practical achievement submissions. |
| AI assistant | Helps draft, simplify, adapt, and refine course content within the approved context. |
| Verifier / reviewer for practical outputs | Reviews submitted proof and approves verified achievements or badges where applicable. |
| Learner preview tester | Tests the course from the learner perspective for clarity, usability, accessibility, and practical relevance. |

## E. Step-by-Step Build Workflow

### Step 1: Open the approved Design-to-Build Handover Record

The creator starts by opening the approved Design-to-Build Handover Record. This should appear as a persistent reference panel inside the Build workspace.

The panel should show:

- course working title;
- target learner group;
- validated capacity gap;
- baseline;
- performance goal;
- required actions;
- approved learning pathway;
- approved block sequence;
- final test requirement;
- practical output or proof requirement, where applicable;
- certificate rule;
- verified achievement or badge pathway, where applicable;
- safeguarding and civic-space requirements;
- accessibility and low-bandwidth constraints;
- AI authoring boundaries;
- evaluation and recognition anchor.
The handover is the anchor, but it should not prevent sensible course-building improvements. Instead, it should guide and validate them.

### Step 2: Create the course shell

The platform generates the basic course shell from the approved handover.

The course shell includes:

- course title;
- course description;
- target learner group;
- estimated duration;
- CSO capacity area;
- relevant indicator or standard;
- language;
- module and lesson structure;
- draft block sequence;
- final test requirement;
- certificate rule;
- practical output or proof submission rule, where applicable;
- verified achievement or badge pathway, where applicable.
The creator may refine the learner-facing title and description for clarity, but should not change the course purpose, capacity area, target learner group, or achievement logic without review.

### Step 3: Use the full block library through the left-side navigation panel

The Build workspace should include a left-side block navigation panel that shows all block types available in the platform.

The left navigation should support:

- expandable main categories;
- visible subcategories;
- search/filter for block types;
- drag-and-drop or “add block” action;
- recommended blocks based on the Design-to-Build Handover;
- indicators showing which blocks are required, optional, or added by creator;
- block-purpose tagging when a block is added.
The block library should be comprehensive enough to support rich course creation, but structured enough to prevent confusion.

### Recommended left-side block navigation structure

| Main category | Subcategories / block types |
| --- | --- |
| 1. Structure & Navigation Blocks | Section header, lesson intro, lesson summary, divider, progress note, next-step instruction, closing block |
| 2. Text & Explanation Blocks | Short explainer, key concept, definition, plain-language note, myth vs fact, do/don’t list, FAQ, glossary item |
| 3. Example & Demonstration Blocks | Worked example, before/after example, annotated example, model answer, common mistake, case snapshot |
| 4. Practice Blocks | Guided practice, template completion, checklist task, drag-and-drop, matching, sequencing, categorization, short response |
| 5. Scenario Blocks | 3C scenario, branching choice, dilemma card, role-based decision, consequence feedback, retry loop, reflection after scenario |
| 6. Reflection & Action Blocks | Reflection prompt, action commitment, personal plan, organizational application prompt, peer discussion prompt |
| 7. Assessment Blocks | Knowledge check, MCQ, true/false, matching quiz, sequencing question, scenario-based question, final test, retake instruction |
| 8. Resource Blocks | Downloadable template, checklist, job aid, worksheet, sample form, offline guide, facilitator note |
| 9. Media Blocks | Image with alt text, compressed illustration, audio note, transcript block, short video, captioned video, infographic |
| 10. Accessibility & Localization Blocks | Translation note, low-bandwidth alternative, transcript, text alternative, plain-language version, printable version |
| 11. Safeguarding & Data Safety Blocks | Safety note, anonymization warning, sensitive data caution, do-no-harm reminder, referral note |
| 12. Practical Output & Recognition Blocks | Proof submission, upload instruction, review criteria, verified achievement note, badge explanation, organization milestone note |

All these blocks may exist in the builder. The difference is that the platform should require each block to have a clear purpose and link.

### Step 4: Build the approved block sequence first

The creator should begin by building the approved blocks from the Design-to-Build Handover.

These are the required blocks that directly come from the approved storyboard or design plan.

Each required block should be linked to:

- required action;
- minimum information;
- practice activity;
- final test preparation;
- final test item;
- practical output;
- recognition pathway;
- safeguarding instruction;
- evaluation anchor.
The platform should visually distinguish:

- Required from Design
- Recommended
- Creator-added
- Needs justification
- Flagged for review

### Step 5: Add additional blocks where they improve the course

The creator may add additional blocks during Build. This is important because real course production often reveals the need for extra learner support.

Additional blocks may be appropriate when they:

- clarify difficult concepts;
- improve learner navigation;
- add a worked example;
- reduce cognitive load;
- support low-literacy users;
- add a needed checklist or job aid;
- provide a low-bandwidth alternative;
- improve accessibility;
- add a safety note;
- prepare learners for the final test;
- strengthen a practical output task;
- add a useful reflection or action commitment.
However, every added block should be tagged with a reason.

### Added block justification options

When adding a block, the creator should select one:

| Justification | Meaning |
| --- | --- |
| Supports required action | Helps learner perform an approved action |
| Provides minimum information | Adds essential knowledge needed for practice or test |
| Improves practice | Adds or strengthens learner practice |
| Improves assessment readiness | Helps prepare for final test or practical output |
| Supports accessibility | Adds transcript, alternative text, simplified wording, or printable version |
| Supports localization | Adds language/context adaptation |
| Supports safeguarding | Adds safety note, anonymization instruction, or do-no-harm guidance |
| Improves navigation | Helps learner understand flow or progress |
| Adds approved resource | Provides template, checklist, job aid, or worksheet |
| Other — requires reviewer note | Needs explanation and may be flagged |

The creator should not add blocks that only expand theory, introduce unrelated topics, or change the course purpose.

### Step 6: Draft learner-facing content using minimum necessary information

The Build Phase converts approved minimum information into learner-facing content.

Content should be:

- practical;
- short;
- clear;
- plain-language;
- locally relevant;
- mobile-friendly;
- easy to translate;
- directly linked to action, test, practice, output, or recognition;
- safe for the learner’s context.
The creator should avoid:

- long theoretical introductions;
- broad background history;
- unnecessary donor jargon;
- generic corporate examples;
- repeated definitions;
- long media scripts;
- content that does not support the final test, practice activity, or practical output.
The Design Phase requires that content be added only after required actions are defined and only when it supports action, assessment, or application.

### Step 7: Use AI-assisted authoring within approved boundaries

AI may support the Build Phase by helping to:

- draft short explanations;
- simplify technical language;
- generate fictionalized CSO examples;
- draft feedback messages;
- convert storyboard notes into learner-facing prompts;
- draft multiple-choice, true/false, matching, sequencing, and scenario-based test items;
- adapt content for low-literacy users;
- create low-bandwidth alternatives;
- suggest plain-language versions;
- draft safety notes based on approved constraints.
AI must not:

- create new course objectives;
- change the target learner group;
- invent new donor requirements;
- provide unsupported legal advice;
- use real beneficiary or sensitive case details;
- create unsafe advocacy instructions;
- add unrelated content;
- ignore safeguarding or civic-space restrictions;
- expand the course beyond the approved scope;
- replace human review.
Every AI-generated block or text item should be reviewed by the creator before it becomes part of the course.

### Step 8: Build Knowledge and Skill learning activities

Phase 1 Build should prioritize Knowledge and Skill learning.

| Gap type | Suitable Build approach |
| --- | --- |
| Knowledge | Short explainer, glossary, checklist, job aid, knowledge check, final test item |
| Skill | Worked example, guided practice, scenario decision, template completion, practical assignment, proof submission |

Motivation and Environment gaps should not be built as Phase 1 courses. If these appear during Build, they should be flagged for non-course routing or future support.

For Skill courses, the platform may still include a final test. The final test confirms key understanding and decision logic. Practical outputs or proof submissions demonstrate application.

### Step 9: Build scenarios, practice activities, and supporting resources

Where the approved design includes practice, the creator should build clear, realistic, locally grounded practice blocks.

Practice activities may include:

- completing a budget justification;
- drafting an advocacy message;
- categorizing community feedback;
- preparing an outcome evidence statement;
- applying a safeguarding referral pathway;
- identifying a conflict-of-interest issue;
- completing a donor compliance checklist;
- preparing a partnership role clarification note.
For scenario-based learning, each scenario should include:

- realistic CSO context;
- learner role;
- challenge or dilemma;
- plausible choices;
- realistic consequences;
- feedback or reflection;
- link to a real CSO action;
- safeguarding or civic-space caution, if needed.
Scenario feedback should explain consequences, not just say “correct” or “incorrect.”

For resources, the creator may add:

- templates;
- checklists;
- job aids;
- worksheets;
- sample forms;
- action planning sheets;
- offline guides.
Each resource should be lightweight and linked to a learner action, final test preparation, practical output, or verified achievement.

### Step 10: Build the final test and automated certificate logic

Every DEC Learning Hub course should include a final test.

The final test may include:

- multiple-choice questions;
- true/false questions;
- matching questions;
- sequencing questions;
- scenario-based decision questions.
The final test should assess essential knowledge and decision logic needed for the approved capacity goal. It should not test unnecessary theory.

### Final test scoring and certificate rule

| Learner result | Platform outcome |
| --- | --- |
| Final test score below 80% | Learner does not pass; retake or review recommended |
| Final test score 80% to 89% | Learner passes the course |
| Final test score 90% or above | Learner receives automated course completion certificate |

The platform should display this distinction clearly:

- 80% and above = course pass
- 90% and above = automated certificate
The final test is part of the standard learning completion pathway for the DEC Learning Hub.

### Step 11: Build the practical output and verified achievement pathway

In addition to the final test, DEC courses may include an action-focused practical output or proof submission.

This is separate from the automated course completion certificate.

A practical output may include:

- completed template;
- uploaded work sample;
- action plan;
- organizational document;
- evidence of applied learning;
- reflection with proof;
- peer-reviewed task;
- mentor-reviewed output;
- community or stakeholder feedback record;
- before/after improvement evidence.
When the learner submits acceptable proof, the platform should allow DEC or an authorized reviewer to recognize the learner or organization with a verified achievement.

This may take the form of:

- badge;
- verified achievement record;
- indicator-linked recognition;
- organizational capacity milestone;
- downloadable proof of achievement;
- donor-facing recognition evidence.
The recognition should be linked to:

- CSO capacity area;
- indicator or standard;
- submitted proof type;
- reviewer decision;
- verification date;
- recognition label;
- organization profile, where relevant.
This is a key distinction:

The final test confirms learning completion. The verified achievement confirms reviewed practical evidence linked to CSO capacity indicators or standards.

### Step 12: Apply safeguarding, civic-space, and data protection checks during production

The Build Phase must apply safeguarding and civic-space requirements from the Design-to-Build Handover.

The course should avoid asking learners to:

- enter real beneficiary names;
- disclose sensitive community data;
- describe active protection cases;
- identify politically sensitive actors;
- share unsafe advocacy details;
- upload documents that expose CSOs or communities to risk.
Where proof submission is required, the platform should instruct learners to:

- anonymize sensitive data;
- remove names and identifying details;
- use fictionalized examples where necessary;
- avoid uploading unsafe documents;
- follow organizational data protection procedures.
Suggested learner safety note:

Do not upload real names, beneficiary details, politically sensitive information, or active safeguarding cases. Use anonymized or fictionalized examples unless your organization has approved the document for sharing.

### Step 13: Apply accessibility, localization, mobile-first, and low-bandwidth requirements

The Build Phase should support learners with different devices, languages, connectivity levels, and accessibility needs.

Build requirements should include:

- short sections;
- readable text;
- clear headings;
- mobile-friendly layout;
- low file sizes;
- compressed visuals;
- no unnecessary heavy video;
- optional audio only where useful;
- downloadable or printable resources where needed;
- alt text for images;
- transcripts for audio or video;
- simple language;
- localization-ready wording;
- inclusive examples;
- low-bandwidth alternatives for media.
The block library should support accessibility and localization as buildable components, not afterthoughts.

### Step 14: Preview the course in learner mode

Before submitting for Review, the creator should preview the course exactly as a learner will experience it.

The preview should check:

- mobile view;
- desktop view, if relevant;
- low-bandwidth mode, if available;
- language display;
- navigation flow;
- required and added blocks;
- interaction behavior;
- final test behavior;
- score calculation;
- certificate trigger at 90% or above;
- practical output submission flow;
- verified achievement pathway, where applicable;
- download links;
- accessibility features;
- completion tracking.
The preview should answer:

Can a busy CSO learner understand what to do, complete the course, take the final test, and submit practical proof where required?

### Step 15: Complete the Build Readiness Check

Before moving to Review, the creator completes a Build Readiness Check.

The check should confirm:

- the course shell is complete;
- approved required blocks are built;
- added blocks are justified;
- all blocks are linked to a valid purpose;
- AI-generated content has been reviewed;
- practice activities work correctly;
- the final test is complete;
- scoring logic works;
- certificate trigger is configured correctly;
- practical proof submission is configured where applicable;
- badge or verified achievement pathway is configured where applicable;
- resources download correctly;
- safeguarding requirements are applied;
- accessibility and localization checks are complete;
- learner preview has been tested;
- evaluation and indicator links are preserved.
If major issues remain, the course should not move to Review.

## F. Platform Design Implications

The Build Phase should appear in the Course Creator Portal as a guided, flexible block-building workspace.

It should not be an unrestricted page builder, but it should also not be a rigid lockstep form. The creator should have access to the full block library through a left-side navigation panel, with the platform providing guidance, linkage, warnings, and review controls.

## 1. Left-Side Block Navigation Panel

The platform should include a persistent left-side panel with expandable block categories and subcategories.

The panel should allow creators to:

- browse all block categories;
- expand and collapse subcategories;
- search for block types;
- see recommended blocks;
- see required blocks from the approved storyboard;
- add additional blocks;
- see which blocks require justification;
- drag blocks into the course canvas or insert them through a plus button.
## 2. Center Course Canvas

The center workspace should show the course structure and learner-facing blocks.

It should support:

- block ordering;
- drag-and-drop reordering where allowed;
- preview of learner-facing flow;
- visual indicators for required vs added blocks;
- missing block warnings;
- block completion status.
## 3. Right-Side Properties and Governance Panel

The right panel should show settings for the selected block, including:

- block title;
- block type;
- linked required action;
- linked practice activity;
- linked minimum information;
- linked final test or practical output;
- accessibility settings;
- safeguarding notes;
- AI drafting options;
- justification for added block;
- review status.
## 4. Full Block Library

The platform should include all major block types, grouped into categories:

- Structure & Navigation
- Text & Explanation
- Example & Demonstration
- Practice
- Scenario
- Reflection & Action
- Assessment
- Resources
- Media
- Accessibility & Localization
- Safeguarding & Data Safety
- Practical Output & Recognition
## 5. Block Purpose Linking

Every block, including creator-added blocks, should be linked to at least one valid purpose:

- required action;
- minimum information;
- practice activity;
- final test preparation;
- practical output;
- accessibility support;
- localization support;
- safeguarding support;
- learner navigation;
- recognition pathway.
## 6. Scope-Control Warnings

The platform should warn, not automatically block, when a creator adds content that may drift from the approved design.

Warning examples:

- “This block is not linked to an approved action or assessment.”
- “This explanation may be too long for mobile learners.”
- “This block introduces a new topic not found in the Design-to-Build Handover.”
- “This proof submission may involve sensitive data.”
- “This media file may not be suitable for low-bandwidth learners.”
## 7. AI-Assisted Authoring Panel

AI should be available inside the Build workspace and should work from the approved handover plus the selected block’s purpose.

AI should support drafting, simplifying, localizing, and refining content, but it should not override the approved design.

## 8. Final Test Builder

The platform should include a final test builder supporting:

- MCQ;
- true/false;
- matching;
- sequencing;
- scenario-based decision questions;
- automated scoring;
- retake settings;
- pass threshold;
- certificate threshold;
- question bank management.
Default rule:

- 80% or above = pass
- 90% or above = automated certificate
## 9. Practical Output / Proof Submission Builder

The platform should allow creators to configure:

- practical output instructions;
- accepted proof types;
- upload or text submission;
- safety and anonymization guidance;
- review criteria;
- reviewer role;
- verification status;
- badge or recognition outcome;
- link to CSO capacity indicator or standard.
## 10. Badge and Verified Achievement System

The platform should support badge or verified achievement recognition for learners or CSOs that submit acceptable practical proof.

This should link to:

- course;
- learner;
- organization;
- capacity area;
- indicator or standard;
- submitted evidence;
- verification decision;
- recognition status.
## 11. Accessibility and Low-Bandwidth Checker

The platform should check for:

- heavy media;
- missing alt text;
- missing transcript;
- long text blocks;
- non-mobile-friendly layouts;
- missing printable alternatives where needed.
## 12. Learner Preview Mode

The creator should preview the actual learner-facing experience, including:

- navigation;
- block sequence;
- added blocks;
- final test;
- score logic;
- certificate trigger;
- practical proof submission;
- verified achievement flow.
## 13. Build Readiness Gate

The Build Phase should end with a readiness gate. The course can move to Review only when required production checks are complete and added blocks are justified.

The Build Phase produces a complete Course Build Draft ready for Review.

Minimum required outputs:

| Knowledge and skill activities | Activities suitable for Phase 1 course scope |

| Course completion certificate | Automatically generated for learners scoring 90% or above |
| Practical output / proof submission | Evidence of applied learning, where applicable |
| Verified achievement / badge pathway | Recognition linked to CSO capacity indicators or standards |
| Downloadable resources | Checklists, job aids, worksheets, templates, or offline support tools |

| Learner preview record | Evidence that the course was tested in learner mode |

# Tool 3.1: Course Shell and Flexible Block Build Checklist

Purpose: Creates the learner-facing course shell, builds the required block sequence, and allows additional blocks through a governed left-side block library.

Output: Built course shell, required blocks, added block register, and block-purpose linkage.

## 1. Purpose of the Tool

The Course Shell and Flexible Block Build Checklist is the first tool in the Build Phase. It converts the approved Design-to-Build Handover Record into the first working learner-facing course inside the DEC Learning Hub Course Creator Portal.

This tool has two connected purposes:

- Governed build: The course must begin from the approved Analysis and Design records, including the capacity gap, performance goal, learner group, required actions, storyboard, final test logic, practical output pathway, safeguarding requirements, and evaluation anchor.

The key rule is:

The creator may add blocks beyond the approved storyboard, but every added block must have a clear purpose and must be linked to the approved course logic.

The output of this tool is a complete course shell, required block sequence, added block register, and block-purpose linkage record ready for AI-assisted content drafting and refinement.

# Tool 3.2: AI-Assisted Content Drafting and Review Log

Purpose: Guides safe use of AI for drafting learner-facing content, examples, feedback, final test items, and block-level refinements while preventing drift from the approved design.

Output: AI-reviewed content log showing what AI produced, what was edited, and what was approved.

## 1. Purpose of the Tool

The AI-Assisted Content Drafting and Review Log is the second tool in the Build Phase. It guides the safe and useful use of AI when drafting, simplifying, localizing, and refining learner-facing course content.

This tool does not allow AI to design the course. The course design has already been approved through the Analysis and Design phases. AI supports the Build Phase by helping the course creator turn approved blocks into clear, practical, mobile-friendly content.

The tool helps DEC ensure that AI-generated content:

- follows the approved Design-to-Build Handover;
- supports Knowledge and Skill gaps only for Phase 1;
- stays linked to approved blocks and required actions;
- uses simple, locally relevant language;
- avoids generic corporate training tone;
- does not invent donor rules, legal advice, indicators, objectives, or examples;
- does not use sensitive or unsafe data;
- is reviewed and approved by a human before entering the final course.
The output of this tool is an AI drafting and review record showing what AI helped produce, what was edited, what was rejected, and what was approved for the course.

# Tool 3.4: Practical Output and Verified Achievement Builder

The tool answers six questions:

- What practical output or proof should the learner submit?
- What CSO capacity indicator or standard is it linked to?
- What safety and anonymization rules must apply?
- Who reviews and verifies the evidence?
- What badge, verified achievement, or organizational milestone may be awarded?
- How should this achievement support the learner’s organization, including donor recognition and capacity-strengthening evidence?
The output of this tool is a configured practical proof submission pathway and verified achievement/badge recognition setup.

## H. Build Phase Quality Standard

A course is ready to leave the Build Phase only when it satisfies this standard:

The course has been built from the approved design, remains linked to the original analysis, focuses only on Knowledge and Skill gaps suitable for Phase 1, uses the full block library appropriately, justifies any added blocks, includes a configured final test, applies the correct pass and certificate rules, includes practical proof or verified achievement logic where applicable, respects safeguarding and accessibility requirements, works in learner preview, and is ready for structured Review.

The Build Phase should preserve the full DEC traceability chain:

Analysis evidence → capacity gap → baseline → root cause → intervention route → action map → learning design → storyboard → flexible block build → final test → certificate / verified achievement → review → publish → evaluate.
