# Output 2 — DEC CSO Capacity Development Framework

## 1. Purpose of the framework

The **DEC CSO Capacity Development Framework** is the logic that connects local CSO capacity evidence to course creation, monitoring, and practical capacity recognition inside the DEC Learning Hub.

It should not be treated as a generic capacity assessment framework or a broad NGO training checklist. Its purpose is to help DEC and its partners move from:

**validated CSO capacity evidence → diagnosed capacity gap → course-fit decision → approved Analysis Record → Design → Build → Review → Publish → learner completion → certificate → optional practical proof → verified achievement → monitoring and course improvement.**

The framework is built from the DEC Learning Hub’s existing workflow logic, especially the requirement that every course should begin from validated CSO capacity evidence and should be linked to capacity areas, standards, indicators, practical gaps, and safeguards. The refined prompt specifically requires this output to connect the nine DEC capacity areas, sub-capacities, standards, gaps, baselines, desired practice, K/S/M/E root causes, course-fit decisions, safeguards, evaluation anchors, Analysis Records, and later workflow stages.

## 2. Core principle: capacity is more than training

For DEC, capacity should be understood as the ability of CSOs, staff, systems, leadership, networks, and enabling environments to perform meaningful functions and improve practice over time. This means capacity is not limited to what an individual knows after taking a course.

The LenCD capacity development resource is especially useful here because it emphasizes that capacity exists across individual, organizational, institutional, and enabling-environment levels, and that capacity is usually a mix of hard and soft components. It distinguishes technical, functional, tangible capacities such as skills, systems, procedures, finance, HR, M&E, and resource mobilization from softer capacities such as culture, leadership, relationships, problem-solving, learning, adaptation, confidence, and participation.

This is important for DEC because the Learning Hub can directly support some capacity needs through digital learning, especially Knowledge and Skill gaps. But it should not claim that a digital course alone can solve leadership incentives, missing organizational systems, unsafe civic-space conditions, weak funding structures, or wider enabling-environment constraints.

The framework therefore applies a disciplined rule:

**Courses are appropriate when the gap is mainly Knowledge or Skill. Motivation and Environment gaps require complementary or non-course support unless a separable Knowledge or Skill component is clearly identified.**

## 3. The DEC capacity-strengthening logic model

The framework can be understood as a traceability chain.

| **Framework layer**           | **Main question**                                                                | **Output**                                                                                                      |
|-------------------------------|----------------------------------------------------------------------------------|-----------------------------------------------------------------------------------------------------------------|
| 1. CSO capacity area         | Which DEC capacity domain is affected?                                           | One of the nine DEC capacity areas                                                                              |
| 2. Sub-capacity              | What specific function or practice is affected?                                  | Specific sub-capacity                                                                                           |
| 3. Indicator / standard      | What standard, commitment, donor expectation, or internal benchmark is relevant? | Indicator/standard reference                                                                                    |
| 4. Capacity gap              | What is not happening well enough?                                               | Validated gap statement                                                                                         |
| 5. Baseline/current practice | What happens now?                                                                | Baseline evidence                                                                                               |
| 6. Desired practice          | What should happen instead?                                                      | Desired practice statement                                                                                      |
| 7. Evidence source           | What confirms the gap?                                                           | Evidence and triangulation record                                                                               |
| 8. K/S/M/E root cause        | Why is the gap happening?                                                        | Knowledge, Skill, Motivation, Environment, Mixed, or unclear                                                    |
| 9. Course-fit decision       | Can a course realistically address this?                                         | Course-addressable, partly course-addressable, non-course route, further diagnosis, or archived                 |
| 10. Intervention route       | What support is needed?                                                          | Course, blended support, mentoring, TA, systems strengthening, safeguarding review, enabling-environment action |
| 11. Safeguards               | What risks must be managed?                                                      | No-harm and visibility rules                                                                                    |
| 12. Evaluation anchor        | What future evidence can show progress?                                          | Monitoring and learning anchor                                                                                  |
| 13. Analysis Record          | What is entered into the platform?                                               | Structured, approved, dashboard-ready record                                                                    |
| 14. Workflow reuse           | How does the record guide course creation?                                       | Design, Build, Review, Publish, Monitoring, certificate, proof, verified achievement                            |

This is the practical “spine” of the DEC Analysis Phase. It keeps the system from asking only, “What training do CSOs want?” and instead asks, “What capacity needs to change, why is it not changing now, and what intervention is realistic?”

## 4. The nine DEC CSO capacity areas

The framework should use the nine DEC CSO capacity areas as controlled categories across Analysis, dashboard views, Design, Build, Review, Publish, and Monitoring.

| **No.** | **DEC capacity area**                                | **Practical meaning for Analysis**                                                                                               |
|---------|------------------------------------------------------|----------------------------------------------------------------------------------------------------------------------------------|
| 1       | Internal Governance and Leadership                   | Board roles, decision-making, leadership accountability, governance integrity, conflict of interest, delegation, meeting records |
| 2       | Transparency and Accountability                      | Information sharing, community accountability, feedback, complaints, stakeholder communication, openness, trust                  |
| 3       | Strategic Planning and Organizational Sustainability | Strategy, mission alignment, operational planning, adaptation, sustainability, organizational risk, continuity                   |
| 4       | Financial Management and Resource Mobilization       | Budgeting, internal controls, donor compliance, financial reporting, fundraising readiness, diversified funding                  |
| 5       | Human Resources, Inclusion, and Safeguarding         | Staff systems, volunteer management, inclusion, gender sensitivity, safeguarding, duty of care, safe practice                    |
| 6       | Evidence-Based Advocacy and Civic Engagement         | Evidence use, safe policy engagement, constituency voice, participation, civic action, advocacy risk management                  |
| 7       | Monitoring, Evaluation, Accountability, and Learning | Outcome monitoring, data use, feedback loops, reporting, learning reviews, adaptive decision-making                              |
| 8       | Digital Skills and Data Use / IT Competencies        | Practical digital tools, responsible data use, information management, online collaboration, data protection                     |
| 9       | Networking, Partnerships, and Collective Action      | Partnerships, coalitions, coordination, peer learning, joint advocacy, shared roles, collective impact                           |

The DEC taxonomy should not be decorative. Annex 4 says it should operate as structured product logic across Analysis, Design, Build, Publish, and Monitoring, including controlled capacity fields, sub-capacity fields, indicator/standard references, evidence sources, baseline/current practice, desired practice, K/S/M/E route, course-fit decision, and dashboard grouping by capacity area.

## 6. How K/S/M/E fits into the framework

K/S/M/E is the root-cause diagnosis layer of the DEC framework. It prevents DEC from treating every capacity gap as a course topic.

| **Route**   | **Definition in DEC Analysis**                                                                                                                                              | **Phase 1 implication**                                                                                   |
|-------------|-----------------------------------------------------------------------------------------------------------------------------------------------------------------------------|-----------------------------------------------------------------------------------------------------------|
| Knowledge   | Learners lack required concepts, standards, definitions, steps, rules, or awareness.                                                                                        | Usually course-addressable.                                                                               |
| Skill       | Learners may know the concept but cannot apply it in a real task, tool, scenario, decision, or work output.                                                                 | Usually course-addressable.                                                                               |
| Motivation  | Learners may know what to do but do not act because of confidence, incentives, leadership signals, ownership, habits, organizational culture, fear, or perceived value.     | Not a Phase 1 course by itself unless a separable K/S component exists.                                   |
| Environment | The barrier is outside the learner: tools, systems, time, staffing, funding, policies, leadership approval, infrastructure, safety, donor rules, or civic-space conditions. | Not a Phase 1 course by itself unless a separable K/S component exists.                                   |
| Mixed       | More than one cause is present.                                                                                                                                             | Proceed only for the explicit Knowledge/Skill component; route other components to complementary support. |
| Unclear     | Evidence is insufficient or contradictory.                                                                                                                                  | Needs further diagnosis before course design.                                                             |

The revised developer-facing description states that the platform should not treat all capacity problems as training problems. It gives the key rule: Knowledge and Skill can proceed to course design; Motivation and Environment should not proceed as Phase 1 courses unless a separable Knowledge/Skill component is identified; Mixed gaps proceed only if the course-addressable component is explicit.

## 7. Course-fit decision logic

After K/S/M/E diagnosis, each Analysis Record should receive a course-fit decision.

| **Course-fit decision**   | **Meaning**                                                                            | **What happens next**                                                                                                                                                                           |
|---------------------------|----------------------------------------------------------------------------------------|-------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| Course-addressable        | The root cause is mainly Knowledge or Skill.                                           | Analysis can proceed to approval and Design handover if other required fields are complete.                                                                                                     |
| Partly course-addressable | A Knowledge/Skill component exists, but Motivation or Environment barriers also exist. | Design can proceed only for the separable K/S component; complementary support is recorded.                                                                                                     |
| Non-course route          | The gap is mainly Motivation or Environment.                                           | Course design remains blocked; issue is routed to mentoring, coaching, leadership engagement, TA, systems strengthening, safeguarding review, resource support, or enabling-environment action. |
| Needs further diagnosis   | Evidence is insufficient, contradictory, or root cause is unclear.                     | Analysis remains open; more evidence or validation is required.                                                                                                                                 |
| Not suitable / archived   | The issue should not become a course or is outside Phase 1 scope.                      | Record may be archived or retained for strategic learning, but not used for course creation.                                                                                                    |

This decision should appear in the final Analysis Record, the native Analysis Dashboard, the Analysis-to-Design Handover, and later read-only context panels in Design and Build.

## 8. Intervention route framework

The framework should make DEC’s response broader than “make a course.” Each gap should have one or more intervention routes.

| **Intervention route**                       | **Use when**                                                                                              | **Relationship to course creation**                                 |
|----------------------------------------------|-----------------------------------------------------------------------------------------------------------|---------------------------------------------------------------------|
| Digital course                               | Knowledge or Skill gap can be addressed through structured learning, scenarios, practice, and final test. | Proceeds to Design.                                                 |
| Course + mentoring                           | Learners need guided application, coaching, or feedback after learning.                                   | Course may proceed, with mentoring noted as complementary support.  |
| Course + template/tool support               | Skill gap requires a job aid, worksheet, checklist, or standard operating tool.                           | Course includes or links to practical tools.                        |
| Technical assistance                         | Gap requires hands-on support to set up systems, policies, templates, or workflows.                       | Course may support awareness/skill, but TA handles system setup.    |
| Leadership or governance engagement          | Motivation or leadership culture blocks practice.                                                         | Course should not be the only intervention.                         |
| Peer learning or cohort support              | Practice improves through exchange, reflection, and accountability among CSOs.                            | Course may be paired with cohort-based learning.                    |
| Systems strengthening                        | Missing procedures, data systems, HR systems, finance systems, or feedback mechanisms block practice.     | Course may be secondary or delayed.                                 |
| Safeguarding / civic-space specialist review | Gap involves protection, advocacy risk, sensitive data, or potential harm.                                | Course proceeds only with strong safety controls or may be blocked. |
| Enabling-environment action                  | Legal, policy, public authority, donor, or infrastructure constraints are primary barriers.               | Not a direct course route except for separable K/S components.      |

This protects DEC from producing attractive but ineffective courses for problems that require organizational or systemic change.

## 9. Safeguards and no-harm layer

Every Analysis Record should include a safeguards review because capacity evidence can expose sensitive organizational weaknesses, civic-space risks, safeguarding issues, or vulnerable groups.

The safeguards layer should check:

| **Safeguards question**                                                                               | **Why it matters**                                      |
|-------------------------------------------------------------------------------------------------------|---------------------------------------------------------|
| Does the gap involve safeguarding, GBV, child protection, or referral issues?                         | Avoid unsafe data collection and unsafe learner tasks.  |
| Does the gap involve advocacy, civic-space, political sensitivity, or public authority relationships? | Prevent exposing CSOs, communities, or rights-holders.  |
| Does the evidence include personal, beneficiary, or community-identifiable data?                      | Require anonymization or avoid collection.              |
| Could dashboard comparison create ranking, shaming, donor pressure, or surveillance?                  | Use safe summaries and role-sensitive visibility.       |
| Could practical proof require risky documents or real cases?                                          | Use simulated, anonymized, or redacted evidence.        |
| Are there gender, inclusion, language, disability, or accessibility concerns?                         | Ensure practical and equitable course design.           |
| Is specialist review required before Design or Publish?                                               | Route sensitive cases to safeguards/civic-space review. |

This aligns with the wider DEC safety principle that learning and capacity evidence should be useful without exposing learners, CSOs, communities, rights-holders, or sensitive organizational information to avoidable risk.

## 10. Evaluation anchor layer

The evaluation anchor is the bridge between Analysis and future Monitoring. It should be defined before course design begins.

Each Analysis Record should include:

| **Evaluation anchor field** | **Purpose**                                                           |
|-----------------------------|-----------------------------------------------------------------------|
| Baseline/current practice   | Shows what is happening now.                                          |
| Desired practice            | Shows what should improve.                                            |
| Evidence confidence level   | Shows how strong the diagnosis is.                                    |
| Potential learning evidence | Shows what the final test might assess.                               |
| Potential practical proof   | Shows what applied evidence could demonstrate later, if enabled.      |
| Monitoring indicator        | Shows what dashboard metric or qualitative signal can track progress. |
| Limitation note             | Prevents overclaiming and identifies what a course cannot prove.      |

For example:

| **Gap**                                                    | **Evaluation anchor**                                                                                                                                                                                         |
|------------------------------------------------------------|---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| Staff cannot write outcome evidence statements.            | Baseline: reports list activities only. Desired practice: reports include evidence of change, source, and confidence. Future evidence: final test scenario + optional outcome evidence worksheet.             |
| Board members do not document conflict-of-interest issues. | Baseline: no consistent record. Desired practice: board minutes include declaration and action. Future evidence: simulated declaration record or approved checklist.                                          |
| Advocacy staff cannot draft safe policy messages.          | Baseline: messages include identifiable community details. Desired practice: messages use anonymized evidence and safe framing. Future evidence: safe advocacy message draft reviewed against risk checklist. |

The monitoring specification for DEC emphasizes that monitoring should connect learner activity back to the original CSO capacity gap, Analysis baseline, Design performance goal, course version, final test, certificate, practical proof, verified achievement, and course improvement decisions.

## 11. Analysis Record as the framework’s central object

The **Analysis Record** is the structured digital object that carries the framework into the platform.

It should include, at minimum:

| **Analysis Record section** | **Core fields**                                                                                                         |
|-----------------------------|-------------------------------------------------------------------------------------------------------------------------|
| Record identity             | Analysis Record ID, CSO/cohort/program context, date, prepared by, status                                               |
| Capacity classification     | capacity area, sub-capacity, indicator/standard, target learner group                                                   |
| Gap definition              | validated capacity gap, baseline/current practice, desired practice                                                     |
| Evidence                    | evidence sources, triangulation status, validation status, confidence level                                             |
| Root cause                  | K/S/M/E route, Mixed-gap details, unclear diagnosis note if relevant                                                    |
| Course-fit                  | course-fit decision, separable K/S component, non-course components                                                     |
| Intervention routing        | digital course, blended support, TA, mentoring, systems strengthening, safeguarding review, enabling-environment action |
| Safeguards                  | no-harm note, sensitivity flags, restricted visibility fields, specialist review requirement                            |
| Evaluation anchor           | future learning evidence, practical proof possibility, monitoring signal, limitation note                               |
| Approval and locking        | reviewer/admin approval, lock status, Analysis-to-Design Handover status                                                |
| Workflow reuse              | Design prefill fields, read-only fields, dashboard visibility, monitoring link                                          |

The refined prompt requires the final Analysis Data Entry Web Form to capture only validated and synthesized information, not every raw field note or interview response, and to produce a comprehensive, standardized, SQL-ready, dashboard-ready, and portal-ready record.

## 12. Relationship to the native Analysis Dashboard

The native Analysis Dashboard should visualize the framework using the same structured Analysis Records. It should not be an external reporting layer. The addendum clearly states that the revised architecture is: field tools, triangulation and validation, final Analysis Data Entry Web Form, SQL database, native Analysis Dashboard inside the DEC Learning Hub platform, and approved Analysis Records reused in Design and later course creation.

The dashboard should help users answer framework-level questions:

| **Dashboard question**                             | **Framework field used**                                   |
|----------------------------------------------------|------------------------------------------------------------|
| Which capacity areas have the most validated gaps? | capacity area, sub-capacity                                |
| Which gaps are course-addressable?                 | K/S/M/E route, course-fit decision                         |
| Which gaps are blocked from Design?                | course-fit decision, status                                |
| Which Mixed gaps have separable K/S components?    | mixed diagnosis, separable component                       |
| Which gaps need complementary support?             | intervention route                                         |
| Which records are ready for Design?                | approval and lock status                                   |
| Which gaps have weak evidence?                     | evidence confidence, validation status                     |
| Which gaps carry safeguards risk?                  | risk flags, no-harm status                                 |
| Which CSO/cohort patterns are safe to compare?     | organization/cohort/region fields with visibility controls |

The addendum also requires the native dashboard to include executive overview, capacity area analysis, K/S/M/E diagnosis, course-fit pipeline, CSO/cohort/region comparison, evidence strength, safeguards/risk flags, Design readiness, and Analysis Record detail views.

## 13. Relationship to later workflow stages

The framework should remain visible across the full Course Creator Portal workflow.

| **Workflow stage**   | **How the framework is used**                                                                                                                                                  |
|----------------------|--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| Analysis             | Defines capacity gap, baseline, desired practice, evidence, K/S/M/E route, course-fit, safeguards, and evaluation anchor.                                                      |
| Design               | Uses approved Analysis Record to create capacity objective, performance goal, Action Map, Learning Design Document, storyboard, assessment intent, and safeguards constraints. |
| Build                | Uses approved Design and Analysis context to build course blocks, scenarios, practice activities, final test, and optional practical proof pathway.                            |
| AI authoring         | AI drafts only within approved Analysis and Design context; it must not invent gaps, evidence, target groups, or K/S/M/E decisions.                                            |
| Review               | Reviewers verify that the course remains aligned with capacity area, indicator, gap, performance goal, safety requirements, assessment logic, and proof pathway.               |
| Publish              | Authorized publisher releases only reviewed and approved course versions with correct metadata and visibility settings.                                                        |
| Learner Runtime      | Learners complete course content and final test. Course remains linked to original capacity gap and target learner group.                                                      |
| Certificate          | Learner receives course certificate at 80%+ final test score; certificate does not claim full organizational transformation.                                                   |
| Practical Proof      | Where enabled, learner/CSO submits safe applied evidence linked to the specific capacity gap or performance goal.                                                              |
| Verified Achievement | Human reviewer verifies practical proof and awards specific capacity-linked recognition or badge.                                                                              |
| Monitoring           | Dashboard aggregates learning, certificates, feedback, proof, verified achievements, and improvement signals by capacity area, course, cohort, and Analysis baseline.          |

The short Course Creator Portal description captures this product identity well: the platform starts from CSO capacity diagnosis, moves through analysis handover, capacity mapping, action mapping, learning design, storyboard, governed building, review, publishing, monitoring, and evidence of capacity strengthening.

## 14. Certificate and verified achievement boundary

The framework must avoid overclaiming.

A course certificate means the learner has passed the final test at the required threshold. It does **not** prove that the CSO has fully transformed its organizational capacity. The broader platform logic confirms that learners scoring 80%+ receive certificates, while practical proof and verified proof may separately lead to badges or capacity achievements.

The distinction should be:

| **Recognition type**                   | **What it means**                                               | **What it does not mean**                            |
|----------------------------------------|-----------------------------------------------------------------|------------------------------------------------------|
| Course certificate                     | Learner passed the final test at 80%+.                          | Does not prove full organizational change.           |
| Practical proof submission             | Learner/CSO attempted to show application.                      | Not yet verified.                                    |
| Verified achievement / badge           | Human-reviewed proof shows a specific applied capacity output.  | Still should not overclaim whole-CSO transformation. |
| Organization capacity evidence summary | Aggregated safe evidence of learning and verified achievements. | Should not become donor surveillance or ranking.     |

This distinction protects DEC’s credibility and CSO trust.

## 16. Practical use standards for DEC

DEC should use this framework with the following standards:

| **Standard**                   | **Practical meaning**                                                                                                      |
|--------------------------------|----------------------------------------------------------------------------------------------------------------------------|
| Start with evidence            | Do not create a course from a topic idea alone.                                                                            |
| Use the nine capacity areas    | Every Analysis Record must map to at least one primary capacity area.                                                      |
| Define the performance problem | Avoid broad topics; write concrete capacity gap statements.                                                                |
| Respect existing capacities    | Do not frame CSOs only by deficits; document strengths and current practice where useful.                                  |
| Diagnose before designing      | K/S/M/E diagnosis must happen before course design.                                                                        |
| Do not force course-fit        | Motivation and Environment gaps should not be disguised as training needs.                                                 |
| Record complementary support   | If a course is only part of the answer, record what else is needed.                                                        |
| Protect sensitive evidence     | Avoid raw sensitive data; use safe summaries.                                                                              |
| Build for monitoring           | Baseline, desired practice, and evaluation anchor must support future learning and improvement.                            |
| Avoid overclaiming             | Certificates show learning threshold; verified achievements show reviewed applied evidence; neither should be exaggerated. |
