# Output 3 — Modular Field Tool Package

## 1. Purpose of this field tool package

This package provides a menu of practical field and consultation tools that DEC staff, consortium leads, consultants, facilitators, and CSO partners can use before platform entry. The tools are designed to help teams gather, compare, validate, and synthesize evidence about CSO capacity gaps before creating a final Analysis Record.

The governing prompt is clear that the Analysis Phase should use a hybrid model: field and consultation tools may be used outside the Course Creator Portal, while the platform should capture only the final validated and synthesized analysis information through one structured Analysis Data Entry Web Form.

This means these tools are not mandatory all-at-once requirements. They should be selected based on the purpose of the assessment, available time, CSO readiness, risk level, and existing evidence.

## 2. How the modular tool package should be used

### 2.1 Use the tools as a menu

DEC or consortium teams may choose:

| Situation | Recommended tool mix |
| --- | --- |
| Very rapid course scoping | Tool 1 + Tool 7 + Tool 9 + Tool 10 + Tool 11 + Tool 12 |
| Full CSO capacity diagnosis | Tools 1–12 |
| Existing assessment already completed | Tool 5 + Tool 8 + Tool 9 + Tool 10 + Tool 11 + Tool 12 |
| Sensitive advocacy/safeguarding topic | Tool 1 + Tool 3 + Tool 5 + Tool 8 + Tool 9 + Tool 10 + Tool 11 + Tool 12 |
| Course idea already proposed | Tool 1 + Tool 5 + Tool 7 + Tool 9 + Tool 10 |
| Multi-CSO cohort analysis | Tool 2 + Tool 3 + Tool 4 + Tool 5 + Tool 6 + Tool 7 + Tool 8 |
| Low-resource / low-time context | Tool 1 + short self-assessment + simplified K/S/M/E worksheet + course-fit tool |

The self-evaluation guide requires this output to treat the tools as a menu, not as a compulsory package, and to ensure every tool maps into the same final Analysis Record structure.

### 2.2 The common data logic

Every tool should help produce one or more of the following Analysis Record fields:

| Final Analysis Record field group | Data the tools help produce |
| --- | --- |
| Capacity classification | capacity area, sub-capacity, indicator/standard |
| Gap definition | validated capacity gap, baseline/current practice, desired practice |
| Evidence | evidence source, evidence type, triangulation status, validation status, confidence level |
| Root cause | Knowledge, Skill, Motivation, Environment, Mixed, or unclear |
| Course-fit | course-addressable, partly course-addressable, non-course route, needs further diagnosis |
| Intervention routing | course, mentoring, coaching, TA, systems strengthening, leadership support, safeguarding review, enabling-environment action |
| Safeguards | sensitivity flags, no-harm note, anonymization need, specialist review requirement |
| Evaluation anchor | possible learner output, means of verification, monitoring signal |
| Handover readiness | ready for data entry, needs more evidence, needs validation, blocked, approved for Analysis-to-Design Handover |

## 3. Tool 1 — CSO Capacity Scoping Template

### Purpose

To define the scope of the Analysis exercise before detailed data collection begins. This tool clarifies which CSO, cohort, capacity area, learner group, and practical problem will be examined.

### When to use it

Use at the start of any Analysis Phase exercise, especially when DEC or a consortium lead is deciding whether a broad issue should become a structured capacity diagnosis.

### Who uses it

DEC capacity lead, consortium lead, analysis consultant, assigned facilitator, or technical expert.

### What data it captures

| Data captured | Maps to final Analysis Record? |
| --- | --- |
| CSO/cohort/program context | Yes |
| Preliminary capacity area | Yes |
| Target learner group | Yes |
| Initial concern or problem statement | Yes, after validation |
| Existing evidence available | Yes |
| Assessment purpose | Yes |
| Risk/sensitivity screening | Yes |
| Proposed tool selection | Indirectly |
| Timeline and responsible persons | Optional/audit field |

### Suggested format

Word template, Google Doc, printable worksheet, or short online form.

### Key sections / questions

| Section | Questions |
| --- | --- |
| Basic scope | Which CSO, cohort, region, or learner group is being assessed? |
| Initial capacity concern | What practical problem triggered this analysis? |
| DEC capacity area | Which of the nine DEC capacity areas is most likely involved? |
| Learner group | Who would need to do something differently if this gap is confirmed? |
| Existing evidence | What documents, reports, feedback, prior assessments, or observations already exist? |
| Sensitivity screen | Does this topic involve safeguarding, civic-space risk, political sensitivity, beneficiary data, or internal vulnerability? |
| Tool selection | Which field tools are necessary and realistic? |
| Decision | Proceed to light analysis, full analysis, specialist review, or defer? |

### Link to final Analysis Record

This tool provides the first draft of the capacity area, target learner group, initial gap statement, evidence source list, and sensitivity flags. These should not be treated as final until triangulated and validated.

## 4. Tool 2 — CSO Self-Assessment Tool

### Purpose

To allow CSO staff or leadership to reflect on their own current practices across selected DEC capacity areas.

### When to use it

Use when DEC wants a broad but practical picture of perceived capacity strengths, weaknesses, and priorities. It is especially useful for multi-CSO cohorts.

### Who uses it

CSO leadership, program staff, finance/admin staff, MEAL staff, safeguarding focal persons, advocacy staff, or relevant functional teams.

### What data it captures

| Data captured | Maps to final Analysis Record? |
| --- | --- |
| Self-rated capacity level | Yes, as one evidence source |
| Current practices | Yes |
| Desired support areas | Yes |
| Perceived causes of gaps | Yes |
| Priority ranking | Yes |
| Examples of current tools or outputs | Yes |
| Confidence level | Indirectly |

### Suggested format

Excel/Google Sheet, Kobo/Google Form, printable checklist, or facilitated group worksheet.

### Recommended rating scale

| Score | Meaning |
| --- | --- |
| 1 | Not yet in place |
| 2 | Started but inconsistent |
| 3 | Basic practice exists |
| 4 | Good practice mostly in place |
| 5 | Strong and consistently used |
| N/A | Not relevant to this CSO or role |

### Key sections / questions

For each selected capacity area:

| Section | Questions |
| --- | --- |
| Current practice | What do we currently do? |
| Practice quality | How consistently do we do it? |
| Evidence | What document, example, or routine shows this? |
| Difficulty | What makes this difficult? |
| Desired practice | What should improve in the next 3–6 months? |
| Learning relevance | Would staff need knowledge, practical skill, tools, coaching, or system support? |
| Priority | Is this low, medium, or high priority? |

### Link to final Analysis Record

Self-assessment results should feed the baseline/current practice, perceived gap, priority level, and evidence source fields. They should not be used alone to approve a course; they must be triangulated with interviews, documents, observations, or validation workshops.

## 5. Tool 3 — Key Informant Interview Guide

### Purpose

To gather deeper explanations from people who understand the CSO’s work, systems, constraints, and capacity challenges.

### When to use it

Use when the analysis needs depth, context, explanation of root causes, or insight into organizational dynamics that cannot be captured through a checklist.

### Who uses it

Analysis consultant, DEC staff, consortium lead, facilitator, or technical expert interviewing CSO leaders, project officers, finance staff, MEAL staff, board members, community-facing staff, partner staff, or reviewers.

### What data it captures

| Data captured | Maps to final Analysis Record? |
| --- | --- |
| Detailed current practice | Yes |
| Concrete examples of gaps | Yes |
| Root-cause explanations | Yes |
| K/S/M/E evidence | Yes |
| Motivation and environment barriers | Yes |
| Safeguarding or civic-space concerns | Yes |
| Existing support or tools | Yes |
| Suggested interventions | Yes |

### Suggested format

Semi-structured interview guide in Word/PDF, audio notes where consented, or typed notes template.

### Key sections / questions

| Section | Sample questions |
| --- | --- |
| Role and context | What is your role, and how do you interact with this capacity area? |
| Current practice | What currently happens when your organization needs to perform this task? |
| Gap evidence | Can you give a recent example where this did not work well? |
| Baseline | What is usually produced now: a report, plan, budget, feedback log, meeting record, advocacy message, or other output? |
| Desired practice | What would good practice look like in your context? |
| Knowledge | What do staff not yet understand clearly? |
| Skill | What can staff not yet do confidently or correctly? |
| Motivation | Are there incentives, confidence, ownership, leadership, or culture issues? |
| Environment | Are tools, systems, time, staffing, funding, internet, policies, or external restrictions blocking practice? |
| Course-fit | What part of this problem could realistically be improved through learning? |
| Safeguards | Are there risks if this issue is discussed, documented, taught, or shown in a dashboard? |
| Evidence | What documents or examples should be reviewed to confirm this? |

### Link to final Analysis Record

Interview findings help complete the root cause summary, K/S/M/E route, course-fit decision, intervention route, and safeguards/no-harm note.

## 6. Tool 4 — Focus Group Discussion Guide

### Purpose

To understand shared experiences, patterns, differences, and practical constraints across groups of CSO staff or stakeholders.

### When to use it

Use when the issue affects a team, role group, cohort, or multiple CSOs and when group reflection can reveal common practice patterns.

### Who uses it

Facilitator, DEC staff, consortium lead, or consultant with a group of CSO staff, program teams, MEAL staff, finance/admin staff, safeguarding focal persons, field staff, or peer CSOs.

### What data it captures

| Data captured | Maps to final Analysis Record? |
| --- | --- |
| Common experiences | Yes |
| Repeated patterns | Yes |
| Differences by role or CSO type | Yes |
| Practical constraints | Yes |
| Learning needs | Yes |
| Motivation/environment issues | Yes |
| Prioritization input | Yes |
| Safeguards concerns | Yes |

### Suggested format

Facilitation guide, flipchart worksheet, Miro/Jamboard board, printable group worksheet, or workshop pack.

### Key sections / questions

| Section | Activity/questions |
| --- | --- |
| Opening | Explain purpose, confidentiality, and no-harm rules. |
| Practice mapping | “Walk us through how this task currently happens.” |
| Pain points | “Where does the process usually break down?” |
| Example collection | “What recent example shows this problem?” |
| Root cause sorting | Group causes under Knowledge, Skill, Motivation, Environment, or Mixed. |

### Link to final Analysis Record

FGDs help confirm patterns, priority, baseline, desired practice, K/S/M/E diagnosis, and intervention route. Sensitive group comments should be summarized and anonymized before platform entry.

To validate synthesized findings with CSO actors and relevant stakeholders before they become final platform records.

### When to use it

Use after initial evidence collection and synthesis, especially before approving priority gaps for course creation.

### Who uses it

DEC capacity lead, consortium lead, facilitator, analysis consultant, CSO representatives, and selected technical or safeguarding reviewers.

### Suggested format

Workshop agenda, slide deck, printable validation matrix, Excel scoring sheet, or facilitated online session.

### Key sections / activities

| Section | Activity |
| --- | --- |

## 9. Tool 7 — Capacity Gap Scoring and Prioritization Matrix

### Purpose

To prioritize which validated gaps should move forward first, especially when several gaps are identified.

### When to use it

Use after initial synthesis and before finalizing Analysis Records or deciding course pipeline priorities.

### Who uses it

DEC capacity lead, consortium lead, M&E/capacity lead, analysis team, CSO representatives, and technical reviewers.

### What data it captures

| Data captured | Maps to final Analysis Record? |
| --- | --- |
| Gap severity | Yes |
| Urgency | Yes |
| Number/type of CSOs affected | Yes |
| Strategic relevance | Yes |
| Course feasibility | Yes |
| Evidence strength | Yes |
| Safeguards risk | Yes |
| Priority level | Yes |
| Recommended next step | Yes |

### Suggested format

Excel/Google Sheet scoring matrix, printable scoring worksheet, or facilitated prioritization board.

### Recommended scoring criteria

| Criterion | Score 1 | Score 3 | Score 5 |
| --- | --- | --- | --- |
| Severity | Minor inconvenience | Affects quality or compliance | Seriously blocks CSO performance or safety |
| Frequency | Rare | Occasional | Common/repeated |
| Reach | One role/CSO | Several teams/CSOs | Many CSOs/cohorts affected |
| Strategic relevance | Nice to address | Supports program goals | Directly linked to DEC priority capacity area |
| Course feasibility | Weak fit | Partial fit | Strong K/S course fit |
| Evidence strength | Anecdotal | Some triangulation | Strong multi-source evidence |
| Safeguards risk | Low | Manageable | High; specialist review required |
| Readiness | Not ready | Needs refinement | Ready for Design handover |

### Priority interpretation

| Total / judgment | Priority category | Recommended action |
| --- | --- | --- |
| High severity + strong course fit + manageable risk | Priority course pipeline | Prepare final Analysis Record |
| High severity + weak course fit | Non-course or blended priority | Route to TA/coaching/systems support |
| High risk + uncertain evidence | Specialist review / more diagnosis | Do not proceed to course yet |
| Low severity + weak evidence | Defer | Archive or revisit later |

### Link to final Analysis Record

This tool feeds priority level, evidence confidence, course pipeline relevance, intervention route, and Design readiness.

## 10. Tool 8 — Triangulation Matrix

### Purpose

To compare findings from different evidence sources and decide whether a capacity gap is sufficiently supported.

### When to use it

Use after collecting evidence from at least two sources, especially before validation and final data entry.

### Who uses it

Analysis consultant, DEC M&E/capacity lead, consortium lead, or technical reviewer.

### What data it captures

| Data captured | Maps to final Analysis Record? |
| --- | --- |
| Evidence source comparison | Yes |
| Repeated patterns | Yes |
| Contradictions | Yes |
| Evidence strength | Yes |
| Confidence level | Yes |
| Data gaps | Yes |
| Need for further validation | Yes |

### Suggested format

Excel/Google Sheet matrix, Word table, or digital synthesis worksheet.

### Template

| Capacity gap candidate | Self-assessment evidence | KII evidence | FGD evidence | Document evidence | Validation evidence | Pattern strength | Contradictions | Confidence level | Next action |
| --- | --- | --- | --- | --- | --- | --- | --- | --- | --- |
| Example: Staff cannot write outcome evidence statements. | Staff rate outcome reporting low. | MEAL lead confirms confusion between outputs/outcomes. | Program staff say donor reports focus on activities. | Reports show activity counts only. | CSO agrees this is accurate. | Strong | Minor | High | Prepare final Analysis Record |

| Confidence level | Meaning |
| --- | --- |

| Medium | More than one source supports the gap, but some detail remains unclear. |
| Low | Evidence is weak, anecdotal, outdated, or contradictory. |
| Not confirmed | Evidence does not support the gap or points to another issue. |

### Link to final Analysis Record

This tool feeds triangulation status, evidence confidence level, evidence summary, contradiction note, and data-entry readiness.

## 11. Tool 9 — K/S/M/E Root-Cause Diagnosis Worksheet

### Purpose

To classify why the capacity gap is happening and determine whether the issue is suitable for Phase 1 course design.

### When to use it

Use after the gap and baseline are defined, before course-fit decision and final data entry.

### Who uses it

Analysis consultant, DEC capacity lead, consortium lead, technical expert, and where appropriate CSO representatives.

### What data it captures

| Data captured | Maps to final Analysis Record? |
| --- | --- |
| Root-cause explanation | Yes |
| K/S/M/E route | Yes |
| Mixed gap components | Yes |
| Separable K/S component | Yes |
| Non-course components | Yes |
| Need for further diagnosis | Yes |

### Suggested format

Worksheet, Excel matrix, form section, or facilitated diagnostic card sort.

### Key diagnostic questions

| Route | Diagnostic questions |
| --- | --- |
| Knowledge | Do learners lack concepts, rules, steps, standards, definitions, or examples of good practice? |
| Skill | Do learners understand the idea but struggle to apply it in a task, template, decision, scenario, or work output? |
| Motivation | Do learners know what to do but lack confidence, incentive, leadership support, ownership, trust, habit, or perceived value? |
| Environment | Are tools, systems, time, staffing, internet, templates, policies, funding, leadership approval, safety, donor rules, or civic-space conditions blocking practice? |
| Mixed | Are several causes present? Which part is course-addressable, and which part is not? |
| Unclear | Is the evidence too weak or contradictory to diagnose? |

### Decision rules

| Diagnosis | Phase 1 course implication |
| --- | --- |
| Knowledge | Can normally proceed to course-fit review. |
| Skill | Can normally proceed to course-fit review. |
| Motivation only | Do not proceed as a course unless a separable K/S component is recorded. |
| Environment only | Do not proceed as a course unless a separable K/S component is recorded. |
| Mixed | Proceed only for the specific K/S component; route M/E components to complementary support. |
| Unclear | Do not proceed; collect more evidence or validate further. |

The developer-facing implementation description requires the Analysis Gate to distinguish K/S/M/E causes and to include a separable Knowledge/Skill component when Motivation, Environment, or Mixed routes are involved.

### Link to final Analysis Record

This tool directly feeds K/S/M/E route, root-cause summary, separable Knowledge/Skill component, non-course component, and Design unlock/block status.

## 12. Tool 10 — Course-Fit Decision Tool

### Purpose

To decide whether a validated capacity gap should proceed into course design, be narrowed into a course-addressable component, or be routed outside the course workflow.

### When to use it

Use after K/S/M/E diagnosis and before approval of the final Analysis Record.

### Who uses it

DEC capacity lead, analysis consultant, consortium lead, technical reviewer, and where needed safeguarding/civic-space reviewer.

### What data it captures

| Data captured | Maps to final Analysis Record? |
| --- | --- |
| Course-fit status | Yes |
| Course-addressable component | Yes |
| Non-course support needs | Yes |
| Recommended intervention route | Yes |
| Design readiness | Yes |
| Blocking reason | Yes |

### Suggested format

Decision tree, checklist, Word/PDF worksheet, or platform form section.

### Decision tree

| Question | If yes | If no |
| --- | --- | --- |
| Is the gap validated by evidence? | Continue | Needs more evidence |
| Is the baseline/current practice clear? | Continue | Revise baseline |
| Is the desired practice clear and realistic? | Continue | Revise desired practice |
| Is the root cause mainly Knowledge or Skill? | Course may proceed | Check M/E/Mixed logic |
| If Motivation/Environment is involved, is there a separable K/S component? | Proceed only for K/S component | Block course; route to non-course support |
| Can a course realistically help learners perform a task, make a decision, use a tool, or apply a standard? | Course-addressable or partly course-addressable | Non-course route |
| Are safeguards risks manageable? | Proceed with controls | Specialist review or block |
| Is there an evaluation anchor? | Ready for handover | Add evaluation anchor |

### Course-fit status options

| Status | Meaning |
| --- | --- |
| Course-addressable | Knowledge or Skill gap suitable for course design. |
| Partly course-addressable | Course can address a defined K/S component, but complementary support is needed. |
| Non-course route | Main issue is Motivation, Environment, or systemic barrier. |
| Needs further diagnosis | Evidence or root cause is unclear. |
| Not suitable for Phase 1 | Out of scope, too risky, or not aligned with DEC Learning Hub course workflow. |

### Link to final Analysis Record

This tool feeds course-fit decision, intervention route, Design readiness, blocking reason, and Analysis-to-Design Handover eligibility.

## 13. Tool 11 — Safeguards and No-Harm Review Checklist

### Purpose

To identify and manage safeguarding, civic-space, data protection, privacy, and no-harm risks before a gap is approved for Design.

### When to use it

Use for every Analysis Record, with deeper review for sensitive topics such as safeguarding, GBV, child protection, feedback/complaints, advocacy, civic engagement, civic-space restrictions, government relations, whistleblowing, politically sensitive contexts, or personal data.

### Who uses it

Analysis lead, DEC capacity lead, safeguarding/civic-space reviewer, data protection focal person, consortium lead, or technical reviewer.

### What data it captures

| Data captured | Maps to final Analysis Record? |
| --- | --- |
| Risk category | Yes |
| Sensitivity level | Yes |
| Unsafe data warning | Yes |
| Anonymization/redaction need | Yes |
| Specialist review requirement | Yes |
| Visibility restrictions | Yes |
| Safe evidence guidance | Yes |
| No-harm note | Yes |

### Suggested format

Checklist, risk-screening form, Word/PDF worksheet, or platform form section.

### Key sections / questions

| Risk area | Review questions |
| --- | --- |
| Safeguarding/protection | Could this involve real cases, children, GBV, referral information, or vulnerable people? |
| Civic-space/political sensitivity | Could the course or evidence expose CSOs, communities, rights-holders, or advocates to retaliation or pressure? |
| Personal data | Does the evidence include names, contact details, photos, case files, complaint logs, or identifiable stories? |
| Organizational vulnerability | Could the dashboard expose weaknesses in a way that creates ranking, shame, donor pressure, or surveillance? |
| Practical proof risk | Would learners be asked to upload unsafe documents or real sensitive evidence? |
| AI safety | Should any evidence be excluded from AI drafting prompts? |
| Dashboard visibility | Should data be aggregated, anonymized, restricted, or hidden? |
| Specialist review | Is safeguarding, civic-space, legal, or data protection review required? |

The revised developer-facing description states that Analysis should capture early safeguards for topics such as safeguarding, protection, advocacy, civic-space restrictions, government relations, sensitive data, beneficiary information, whistleblowing, and politically sensitive contexts, and that the safeguards/no-harm note should follow the course into Design, Build, Review, and Monitoring.

### Risk level options

| Risk level | Meaning | Required action |
| --- | --- | --- |
| Low | No sensitive data or obvious risk | Standard no-harm note |
| Moderate | Some sensitivity, manageable with anonymization or wording controls | Add safeguards note and visibility limits |
| High | Safeguarding, civic-space, political, or personal data risk | Specialist review required before Design |
| Do not proceed | Risk cannot be safely managed in course format | Block course or redesign analysis approach |

### Link to final Analysis Record

This tool feeds risk flags, safeguards/no-harm note, sensitivity level, visibility restriction, specialist review status, and safe evidence guidance.

## 14. Tool 12 — Evaluation Anchor Worksheet

### Purpose

To define what future evidence can show whether the course helped learners move toward the desired practice.

### When to use it

Use after the gap, baseline, desired practice, and course-fit decision are defined, before final Analysis Record approval.

### Who uses it

M&E/capacity lead, analysis consultant, DEC capacity lead, course design lead, or technical expert.

### What data it captures

| Data captured | Maps to final Analysis Record? |
| --- | --- |
| Baseline/current practice | Yes |
| Desired practice | Yes |
| Potential final test focus | Yes |
| Possible learner output | Yes |
| Practical proof possibility | Yes |
| Monitoring signal | Yes |
| Limitations/overclaiming note | Yes |

### Suggested format

Worksheet, Excel table, Word template, or form section.

### Key sections / questions

| Section | Questions |
| --- | --- |
| Baseline | What do CSO staff currently do or produce? |
| Desired practice | What should they be able to do better after learning and support? |
| Minimum evidence of learning | What should the final test check? |
| Practical output | Could learners produce a worksheet, template, decision, plan, message, or checklist? |
| Practical proof suitability | Is there safe applied evidence that could later be submitted for review? |
| Monitoring signal | What dashboard metric or qualitative feedback should be tracked? |
| Limitation | What should DEC not claim from the course or certificate alone? |

### Example

| Field | Example entry |
| --- | --- |
| Gap | Program officers cannot write outcome evidence statements. |
| Baseline | Reports mostly list activities and participant numbers. |
| Desired practice | Officers can write a short outcome evidence statement using one indicator and one evidence source. |
| Final test focus | Distinguish output vs outcome; select strongest evidence statement. |
| Possible learner output | Completed outcome evidence worksheet. |
| Practical proof possibility | Yes, if worksheet is anonymized and uses non-sensitive project data. |
| Monitoring signal | Final test pass rate, certificate count, worksheet submissions, verified achievement count, learner feedback. |
| Limitation note | Certificate shows learning threshold, not full organizational MEAL transformation. |

The Analysis Gate should create this first evaluation anchor so that later monitoring connects back to the original capacity gap, baseline, desired practice, and evidence logic.

### Link to final Analysis Record

This tool feeds evaluation anchor, possible learner output, practical proof possibility, monitoring signal, and limitation/overclaiming note.

## 15. Cross-tool synthesis rule

The tools should not produce separate disconnected reports. They should feed one shared synthesis process:

| Evidence source | What it contributes | Must be checked before final entry |
| --- | --- | --- |
| Scoping template | Initial problem framing | Is the scope too broad or vague? |
| Self-assessment | CSO perception of strengths/gaps | Is it triangulated with other evidence? |
| KII | Root-cause depth | Are individual views overgeneralized? |
| FGD | Shared patterns and role differences | Were quieter or sensitive voices protected? |
| Document review | Existing evidence | Is the evidence current and relevant? |
| Validation workshop | Stakeholder confirmation | Was validation meaningful, not tokenistic? |
| Prioritization matrix | Priority and feasibility | Did high-risk gaps receive safeguards review? |
| Triangulation matrix | Evidence confidence | Are contradictions documented? |
| K/S/M/E worksheet | Root-cause route | Is the separable K/S component explicit where needed? |
| Course-fit tool | Design eligibility | Is training being overused? |
| Safeguards checklist | Risk controls | Are unsafe details excluded from platform entry? |
| Evaluation anchor worksheet | Future monitoring logic | Is the course outcome not overclaimed? |

## 17. Practical packaging recommendation

For implementation, DEC can package the tools into three levels:

| Package level | Contents | Best use |
| --- | --- | --- |
| Light / rapid package | Scoping template, short self-assessment, document review, K/S/M/E worksheet, course-fit tool, safeguards checklist, evaluation anchor | Rapid validation of a course idea |
| Standard package | Scoping, self-assessment, KIIs, document review, triangulation, prioritization, K/S/M/E, course-fit, safeguards, evaluation anchor | Most course pipeline analysis |
| Deep / cohort package | All 12 tools, including FGDs and validation workshop | Multi-CSO cohort diagnosis or high-priority capacity area |

This prevents the toolkit from becoming too heavy for local CSOs while keeping it rigorous enough for evidence-based course creation.
